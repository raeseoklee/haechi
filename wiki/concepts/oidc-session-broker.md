# OIDC Session Broker

`haechi-auth-oidc@0.1.0` (`satellites/auth-oidc/`) is an **interactive OIDC session broker** — the authorization-code + PKCE flow that lets a human log in through a browser and obtain a server-side session. It is the [[dashboard-audit-viewer]]'s login mechanism and the **interactive-auth half** that [[identity-and-auth]]'s `haechi-auth-jwt` (a per-request *bearer* validator) left open. Shipped in the 0.9 cut (`docs/current/release-0.9-implementation-scope.md` §2.2; PR #40).

Exports `createOidcSessionBroker(options)` + `normalizeOidcConfig(options)` (+ injectable in-memory store factories). It is **not** an `authProvider` — `createJwtAuthProvider` keeps the per-request role. The broker returns an object satisfying the dashboard `sessionGuard` contract `{ authenticate(req), handlers: { "/auth/login", "/auth/callback", "/auth/logout" } }`, so wiring is pure injection (`createDashboardServer({ sessionGuard: createOidcSessionBroker({ ... }) })`).

## Config (fail-closed)

`normalizeOidcConfig` enumerated-throws on every option, rejects unknown keys. Required: `cryptoProvider` with `hmac()` (no PII-safe identity / `sessionIdHash` without it), `issuer` (https URL), `clientId`, `redirectUri` (https or loopback-http carve-out, **same-origin** with the broker, **path === `/auth/callback`** — the identical `redirect_uri` is sent on both authorize and token exchange per RFC 6749). `clientSecret` optional (omitted ⇒ public PKCE-only client). `scopes` force-include `openid` (deduped) and **strip `offline_access`** (refresh out of scope). `returnToAllowlist` relative same-origin paths. `tokenEndpointAuthMethod` `client_secret_basic` (default) | `client_secret_post`, **never `none`** for a confidential client. Cookie hardening (`secureCookies` true|false|`'auto'`, `trustProxy`) keys off the **externally-visible** scheme — off-loopback without confirmed https fails closed. TTLs (`sessionTtlSeconds`/`idleTtlSeconds`/`maxAgeSeconds`), `pendingCap`, `rateLimitMax`, and injectables (`fetchImpl`/`lookupImpl`/`now`/`sessionStore`/`pendingStore`/`auditSink`) round it out.

## Flow security (each an acceptance criterion)

- **`GET /auth/login`** — CSPRNG `state`/`nonce`/`code_verifier`; `code_challenge = S256(code_verifier)` (**S256 mandatory, never `plain`**); persist the trio + the pinned resolved `issuer`/`token_endpoint`/`jwks_uri` in a server-side **pending-auth** record keyed to a short-TTL `__Host-haechi_preauth` cookie; `302` to the discovered `authorization_endpoint`. `max_age` sent when configured.
- **`GET /auth/callback`** — **state-first short-circuit**: atomically `take()` the pending record by the pre-auth cookie and constant-time-compare `state` **before any outbound request**. A missing/used/mismatched state or missing pre-auth cookie → identical generic `401` deny with **no IdP round-trip** (defeats authorization-code injection / login-CSRF and the replay TOCTOU). Then redeem `code` **only at the pinned `token_endpoint`** with `code_verifier`; verify the ID token via the shared verifier (+ `nonce` match) and check any RFC 9207 `iss` (callback param and token-response) equals the pinned issuer (**mix-up defense**); **mint a fresh CSPRNG session id** (no fixation), discard the pre-auth cookie, set the session cookie, `302` to an **allowlisted relative** `return_to` (default `/`).
- **`POST /auth/logout`** — **non-GET, CSRF-gated** by a required `X-Haechi-CSRF` custom header (proves a same-origin fetch; `connect-src 'self'` enforces it — not `SameSite` alone). Fully destroys server-side session state (replaying the old cookie → `401`), clears the cookie. Optional RP-initiated logout to an **allowlisted** `post_logout_redirect_uri` only (off-allowlist → `400`, no open-redirect).

## Discovery and egress (SSRF-hardened)

- Fetch `<issuer>/.well-known/openid-configuration` over https, bounded body (≤ 1 MiB); **reject unless `metadata.issuer` string-equals the configured `issuer`** (issuer-confusion guard) and pin the verifier's expected `iss`. **Single-origin only (0.9):** `authorization_endpoint`/`token_endpoint`/`jwks_uri`/`end_session_endpoint` must share the issuer hostname — same constraint as `haechi-auth-jwt` 0.8.
- **Every egress runs the same guard, not just JWKS.** The authorization-code flow adds a `token_endpoint` POST the bearer verifier never makes; a token endpoint that DNS-rebinds to `169.254.169.254` between discovery and exchange is the classic metadata-exfil path. So discovery GET, JWKS GET (via the verifier), token-exchange POST, and any end-session redirect each run a `lookup`-then-`isBlockedAddress` **post-DNS re-check immediately before the request**, with `redirect: "error"`, a bounded body, and a fetch timeout. The broker **reuses `isBlockedAddress` exported from `haechi-auth-jwt`** (the auth ecosystem shares one copy; the `haechi-crypto-kms` Vault backend deliberately keeps its own — see [[packaging-and-distribution]]).

## Shared JWS verifier + ID-token profile

0.9 refactors `haechi-auth-jwt@0.2.0` to **additively export `createJwtVerifier`** — a standalone primitive carved out of the existing `resolveJwk`/`verifySignature`/claim-validation, **behavior-preserving**. `createJwtAuthProvider` is reimplemented on it (all 0.8 tests pass unchanged) and still owns Bearer-header parsing. The primitive verifies signature + `alg`/`kid`/RSA-bits + `iss`/`aud`/`exp`/`nbf` (the exact 0.8 surface); **`nonce` is NOT baked in** — `verify(jwt, { expectedNonce })` checks it only when the caller passes it (a no-op for bearer). The broker peer-depends on `haechi-auth-jwt >=0.2.0 <1.0.0` and layers an **OIDC ID-token `aud`/`azp` profile** on the verified claims (distinct from the lenient bearer `aud`): `aud` MUST contain `clientId`; a multi-valued `aud` requires `azp` present and `azp === clientId`; a single `aud` must equal `clientId` (OIDC Core §3.1.3.7). The broker is a **pure-login consumer** — it **discards the access token**, which makes `at_hash`/`c_hash` validation intentionally out of scope. This gives the ecosystem exactly **one** audited JWS/JWKS verification path.

## Sessions and identity

Server-side sessions; the cookie carries **only** an opaque ≥256-bit CSPRNG id — ID/access/refresh tokens and the `client_secret` never reach the browser or a log. Two hardened cookies (`__Host-haechi_preauth` single-use + `__Host-haechi_session`), both `HttpOnly` + `SameSite=Lax` (Lax — Strict would drop the IdP→callback top-level GET) + `Secure` + `__Host-` when the external scheme is https (an off-loopback-plaintext opt-in drops the prefix). Default in-memory `sessionStore`/`pendingStore` with an **atomic `take()`** (consume-and-delete) contract; absolute + idle TTL eviction; a **hard pending-auth cap** that rejects new logins with a generic `429` rather than evicting an in-flight auth. Identity is built via core's `buildExternalIdentity` (keyed-HMAC `subjectHash`, domain `haechi:identity:hash:v1`, `provider: "oidc"`); raw `sub`/email/name never logged or stored.

## Audit trail (PII-safe)

The broker takes an injectable `auditSink` and emits `oidc.login.start` / `oidc.login.success` / `oidc.login.failure{reasonCode}` / `oidc.logout` / `oidc.session.evict`, each carrying **only** keyed-HMAC `subjectHash`/`issuerHash`/`sessionIdHash` (domain `haechi:oidc:session-id:hash:v1`), `provider:"oidc"`, a coarse `reasonCode` enum (`state_mismatch|nonce_mismatch|token_invalid|exchange_failed|host_blocked|expired`), and a timestamp — so brute-force against `/auth/callback` is **visible** (a per-request validator could omit this; an interactive login can't). Events go through the broker's own allowlist projection. The one core touch is additive: `packages/audit` `FORBIDDEN_KEYS` gains `access_token`/`id_token`/`refresh_token`/`code`/`code_verifier`/`client_secret`/`state`/`nonce`/`sub`/`email` (no behavior change to existing events).

## Packaging

`peerDependencies: { haechi: ">=0.8.0 <1.0.0", "haechi-auth-jwt": ">=0.2.0 <1.0.0" }` — core peer stays `>=0.8.0` (it uses only `buildExternalIdentity`, present since 0.6/0.8; not over-tightened to `>=0.9.0`), auth-jwt peer is `>=0.2.0` for the new verifier export. **Zero runtime dependency** (`node:` `fetch`/`crypto`/`dns` suffice), `publishConfig: { access: "public", provenance: true }`. Publishes via `.github/workflows/auth-oidc-publish.yml` on an `auth-oidc-v<semver>` tag. See [[identity-and-auth]] and [[packaging-and-distribution]].
