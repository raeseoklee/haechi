## [2026-06-10] design | 0.8 ecosystem-foundation design finalized

Detailed-design pass for 0.8 (no code), hardened by two adversarial-review workflows (43 + 38 agents) and empirical testing of the linchpin assumptions. Scope: npm workspaces monorepo + first two published satellites `haechi-crypto-kms` (real AWS KMS) and `haechi-auth-jwt` (headless JWKS); `@haechi/auth-oidc` + `@haechi/dashboard` split to a new 0.9 row. Key empirical findings that reshaped the draft: (1) the obvious workspaces layout (`["satellites/*"]` + satellite peer range) silently fails — npm resolves the peer from the registry (`ETARGET`) and never links root; the working layout is root self-member `["."]` + satellite peer+dev dual core dep (verified: clean install, symlink, no satellite leak, `node --test` traverses satellites without a symlink-cycle hang). (2) JWS ES256 signatures are raw R‖S but `node:crypto.verify` defaults to DER and returns `false` — silently rejecting every valid ES256 token; spec now mandates `dsaEncoding: "ieee-p1363"`. The auth-jwt §2.4 security spec pins concrete constants (RSA ≥2048, clock-skew cap 300s, kid-flood cooldown 60s, HMAC-SHA-256 identity, 1 MiB JWKS bound, single-origin issuer SSRF guard). The zero-dep gate inspects the packed manifest (not the installed-tree SBOM, which passes vacuously) and is negatively tested. Wrote `release-0.8-implementation-scope.md` (+KO), updated `[[release-roadmap]]` (new 0.9 row), `[[packaging-and-distribution]]` (verified workspaces mechanic), and the risk-register §8 (EN+KO). ## [2026-06-11] review+fix | real-environment validation (vLLM + Ollama) and response false-positive fix

Validated the proxy live against a real vLLM (Qwen3.6-35B @ 10.0.0.50) and a real Ollama (10.0.1.55). Core behaviours held on both adapters: protect-then-forward with the model provably receiving `[TOKEN:…]` (echo-proven on vLLM), card→block (403), the Ollama implicit-streaming fail-closed invariant (`/api/chat` without `stream:false` → 501), no plaintext in audit, and the transparent token round-trip (caller gets the original back). Real testing surfaced a response-direction false-positive class that 502-blocked legitimate completions under `responseProtection: enforce`: the unix-timestamp `created` matched the KR phone rule, and the echoed `[TOKEN:…]` matched the secret rule (`TOKEN:` reads like a `token:<secret>` assignment — Haechi blocking its own token). Fixed in `packages/filter/index.mjs`: phone rule rejects bare separator-less non-`0`-led digit runs; detection skips Haechi's own `[TOKEN:…]`/`[HAECHI_ENC:…]`/`[REDACTED:…]` markers **on the response direction only** (request-side stays full-scan, so a fake marker can't smuggle a secret past request detection — caught during self-review before shipping). Verified the fix end-to-end against real vLLM (round-trip now 200 under enforce, caller gets the restored email). Added `tests/filter.test.mjs` cases (marker-not-re-detected, request-side-no-bypass, phone-ignores-timestamps); 203 tests / 201 pass. Docs: CLAUDE.md invariant, threat-model (EN+KO) two new rows, configuration.md (EN+KO) `report-only`-for-real-upstreams recommendation. New page [[2026-06-11-real-environment-validation]]. Residual (documented): a Luhn-passing long numeric field can still match `card` on the response → prefer `responseProtection.mode: report-only` for real upstreams.
