## [2026-06-11] design | 1.1 scope pinned (process-isolated capability enforcement)

Authored `docs/current/release-1.1-implementation-scope.md` (+ `.ko`) after a maintainer scoping interview (4 recommended choices: process-isolated + permission model / new strong mode alongside the unchanged worker / host-mediated fetch for credential containment / focus 1.1 on capability enforcement) and a 3-lens adversarial security review with **empirical Node-26 testing** — which reshaped the design. 1.1 closes the most-cited **1.0 honest residual**: `worker_threads` is memory/crash isolation, NOT a capability sandbox, so a malicious *signed* plugin could still use fs/net + exfiltrate its credential. The new opt-in **`process-isolated`** runtime (`createProcessIsolatedAuthProvider`) runs a signed `authProvider` in a child `node` under `--permission` with **zero grants**, loaded from a `data:` URL (no fs grant → no TOCTOU/symlink/realpath surface), `stdio:['ignore','ignore','ignore','ipc']` (the review's key catch — a child adds stdout/stderr/fd write channels `--allow-net` doesn't gate), scrubbed env, JSON-string-only IPC. **Three critical first-draft errors the empirical review caught + fixed:** (1) a "delete `node:net`/fetch" harness is NOT containment — `process.binding('tcp_wrap')` opens a live socket and `import('node:net')` re-resolves a fresh builtin past any cache deletion; real net containment is the kernel-enforced **`--allow-net` denial**, and since Node 22 LTS has no `--allow-net`, the default **`netEnforcement:"require-permission"`** *fails closed* (throws at construction) rather than pretending; (2) the child's stdio is a leak channel → all ignored; (3) feature detection was unspecified → now `process.allowedNodeEnvironmentFlags.has('--allow-net')` + a spawn-probe, fail-closed. Credentials: the **host** runs the audited `createJwtVerifier` for standard JWT (no plugin sees it); `process-isolated` is for **custom/opaque** credentials, contained by net+stdio+fs denial; key material is **host-fetched from operator-declared URLs** (no plugin-driven SSRF) — which **promotes `isBlockedAddress`/`guardedFetch` into a core node:-only module** (core stays zero-dep; satellites re-import, ending the drift). Lifecycle audit gains **host-computed/enum-only** fields (`isolation`/`grants`/`netEnforcement`) outside the frozen protect-event schema (the 1.0 `api-contract.test.mjs` is unaffected). Anti-DoS: warm long-lived child + respawn-storm circuit breaker + exponential backoff. Proposed risks **P1-SEC-026** (process-isolated capability enforcement, strengthens P1-SEC-024) / **P1-SEC-027** (host-mediated key material + core SSRF guard). 3-PR stack (process runtime / credential containment + core SSRF guard / feature-detect + lifecycle + 1.1.0 cut). Updated `[[release-roadmap]]` with the 1.1.0 row. Implementation deferred to PRs.

## [2026-06-11] review+fix | real-environment validation (vLLM + Ollama) and response false-positive fix

Validated the proxy live against a real vLLM (Qwen3.6-35B @ 10.0.0.50) and a real Ollama (10.0.1.55). Core behaviours held on both adapters: protect-then-forward with the model provably receiving `[TOKEN:…]` (echo-proven on vLLM), card→block (403), the Ollama implicit-streaming fail-closed invariant (`/api/chat` without `stream:false` → 501), no plaintext in audit, and the transparent token round-trip (caller gets the original back). Real testing surfaced a response-direction false-positive class that 502-blocked legitimate completions under `responseProtection: enforce`: the unix-timestamp `created` matched the KR phone rule, and the echoed `[TOKEN:…]` matched the secret rule (`TOKEN:` reads like a `token:<secret>` assignment — Haechi blocking its own token). Fixed in `packages/filter/index.mjs`: phone rule rejects bare separator-less non-`0`-led digit runs; detection skips Haechi's own `[TOKEN:…]`/`[HAECHI_ENC:…]`/`[REDACTED:…]` markers **on the response direction only** (request-side stays full-scan, so a fake marker can't smuggle a secret past request detection — caught during self-review before shipping). Verified the fix end-to-end against real vLLM (round-trip now 200 under enforce, caller gets the restored email). Added `tests/filter.test.mjs` cases (marker-not-re-detected, request-side-no-bypass, phone-ignores-timestamps); 203 tests / 201 pass. Docs: CLAUDE.md invariant, threat-model (EN+KO) two new rows, configuration.md (EN+KO) `report-only`-for-real-upstreams recommendation. New page [[2026-06-11-real-environment-validation]]. Residual (documented): a Luhn-passing long numeric field can still match `card` on the response → prefer `responseProtection.mode: report-only` for real upstreams.

## [2026-06-11] fix | response-direction skips bare JSON number leaves (close the card/RRN metadata residual)

Follow-up to the real-environment validation: closed the documented residual where a long Luhn-passing numeric metadata field (Ollama emits nanosecond `*_duration` fields; a long generation can reach 13–19 digits) false-positives as `card`, and a 13-digit one as `kr_rrn`. Evidence-first: confirmed a 16-digit number leaf → `card` and that `entry.kind === "number"` distinguishes number leaves. Fix in `packages/filter/index.mjs` detectEntry: on `direction === "response"`, skip detection on number-kind leaves entirely (parallel to the marker exclusion). Rationale: response numbers are inference-server metadata, never a model-leaked card/phone/RRN; a real model leak lands in generated TEXT (string leaves), which are still inspected; the REQUEST direction still scans numbers (a client can send a card as a number — verified unchanged). Tests added (response number-leaf not scanned; response-text card still caught; request number card+RRN still caught). 206 tests / 204 pass. Real vLLM + Ollama responses now scan with zero detections under enforce. Updated CLAUDE.md, threat-model + configuration (EN+KO), and [[2026-06-11-real-environment-validation]].
