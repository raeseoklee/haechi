The hard part was routing: Gemini is **model-in-path with a `:method` suffix** (`/v1beta/models/{model}:generateContent`), but `matchRoute` did EXACT pathname match. **Additive fix:** `matchRoute` is now two-phase — phase 1 the original exact `candidate.path === pathname` (returned immediately, so every existing adapter classifies byte-identically), phase 2 (only when no exact hit) matches `pathname.endsWith(":"+route.methodSuffix)`. Request/response protection rides the core tree-walk (systemInstruction.parts[].text + contents[].parts[].text + candidates[].content.parts[].text all walked — no custom extraction). `?alt=sse`/`?key=` query strings are stripped by `pathFromRequestUrl` so the suffix still matches; the client's `x-goog-api-key` is forwarded. ## [2026-06-12] ingest | Opt-in base64/percent-decode detection rescan (closes the WS2d encoding residual)

Closes the documented WS2d residual: a card/RRN/secret base64- or percent-encoded before sending previously passed every rule (Haechi matched NFKC text but never decoded). New **opt-in** `filters.decodeAndRescan` (boolean, default `false`, fail-closed-validated in `normalizeConfig`, threaded via `createDefaultFilterEngine(normalized.filters)` into `detectEntry`): when on, after the normal scan each string leaf is decode-and-rescanned. **Why opt-in + precision-guarded:** WS2d deferred this because an always-on decode is FP-prone. Three stacked FP guards make it precision-neutral: (1) `decodeLeaf` only attempts a leaf that LOOKS encoded — anchored base64/base64url alphabet + valid length-mod + 16–8192 bounds, or a `%XX`-bearing string for percent; (2) a **round-trip guard** (`bytes.toString(encoding) !== value` → reject, since `Buffer.from` is lenient) + an **`isUtf8`** byte check (random base64 → non-UTF-8 → skip before rules run); (3) `isDecodeEligibleRule` restricts a decoded hit to **hard-block types or checksum-validated rules** (`phone` excluded — its `validate` is a trunk heuristic, not a checksum), so a phone-shaped run in random decoded bytes can't fire. A decoded hit can't map to an original offset, so it emits a **fail-closed WHOLE-LEAF detection** (reusing the WS2d Case-3 `wholeLeafDetections`, now taking a `ruleFilter`) — the entire encoded leaf is redacted/blocked. **Opt-in stays OUT of the default corpus** (`scan:detection` measures the default engine; a base64 positive would be an FN) — proven by dedicated `tests/decode-rescan.test.mjs` (16) with the flag on. Built via a workflow (implement → adversarial verify, PASS/breaches []); re-verified by hand: 5,000 random base64 → 0 FP, benign prose/blob/JWT → 0 *decode-introduced* FP (the JWT's `secret` hit is the native `jwt` rule, identical flag on/off — a true positive, not a decode FP — initially mis-flagged by my own harness until I compared on-vs-off), flag-off byte-identical, encoded card→whole-leaf redacted end-to-end, decoded soft-phone suppressed. Docs: threat-model(+ko) base64/percent moved from a hard exclusion to an opt-in control; configuration(+ko) + CLAUDE.md + example config. Zero new runtime dep (`node:buffer` `isUtf8`/`Buffer` + `decodeURIComponent`); api-contract green (additive key); preflight green (624 tests / 622 pass / 2 skip). Verified the fix end-to-end against real vLLM (round-trip now 200 under enforce, caller gets the restored email). Added `tests/filter.test.mjs` cases (marker-not-re-detected, request-side-no-bypass, phone-ignores-timestamps); 203 tests / 201 pass.
